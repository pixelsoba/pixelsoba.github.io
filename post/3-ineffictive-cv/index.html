An inefficient and impractical way to present a resume

Jun 21, 2021 · Augustin Ambiehl

Lately, I’ve been experimenting a lot with Godot GUI to see what it’s capable of and to get my hands on it. What better way to do that than to start a small project? Nostalgic for Windows XP, I wanted to make a web export of a Godot project that mimics as well as possible the rendering of this Windows of 2001. And with the objective to create the biggest possible gap between these two universes which do not mix much: dev web & dev desktop/software. For that, I tried to make a website with a user experience as close as possible to a desktop navigation.

Obviously, to offer a complete experience on a base as large as an OS is totally out of reach. So I had to restrict the scope to what I thought was essential to have a beginning of feeling Windows XP.

  • The Startup menu, with a convincing and dynamic management of the items it contains.
  • The Desktop with some icons, and the possibility to select some with a dotted rectangle.
  • Windows that resize, move, shrink or go full screen.
  • Remaking the mythical 3D Pinball SpaceCadet …unfortunately, I don’t have weeks to spend extracting assets and recoding this game üòÖ. Even if there was no lack of desire, I had to settle for a small cameo.
  • Slip in my CV to convince me that I’m not doing all this for nothing.

Challenges encountered

Find the graphics

My first mistake was to underestimate the recovery time of the graphic elements… Since I couldn’t find an online library complete enough for what I needed, I had to dig through old tutorials captured in HD and hope to find elements in a conformation that suited me.

Web export optimization

As I had the constraint to make my application work on as many web browsers as possible, I chose the maximum compatibility with OpenGLES 2 (aka WebGL 1). Fortunately, Godot offers an excellent abstraction of the graphics API, so this is not a big problem. The main issues were flags for some textures that did not allow a used version of the textures, but only a stretched version.

For example, in the case of the selection rectangle on the desktop, since GLES2 does not allow convincing tiling of a dotted line texture, I had to look into an algo that draws each dotted line using draw_lines. This solution is more elegant, but it took much more time to set up.

I found that the mobile supported my application very well. However, I didn’t anticipate that the management of touch events is not totally identical to that of classic mouse events. I was still able to make touch work by allowing touchEvents on elements that previously worked only with mouseRelease. The use of touch and release to interact with the ihm is not really instinctive, that’s why I only used incoming/pressing events for the touch part.

Conclusion

It’s really a pleasure to make interfaces with Godot, the Control node system is really well designed (I talk about it on this post GODOT: 9 TIPS FOR CREATING INTERFACES). The editor is still fast, light and powerful. For projects of this magnitude, I don’t know of a more efficient solution.

This is also the first time I’ve forced myself to work with GLES 2 for web support. Even though GLES 3 (in WebGL 2) works for the most part on the latest browsers, it is not necessary for my purposes. Using Godot with GLES 2 for a web target is no different than using it in the traditional way, except that you have to touch up the HTML-JS template a bit if you want a more specific integration.

I also paid attention to the size of my web export and to the optimization of the size of my assets. My final export is composed of two very large files compared to what we usually see in the web world: a binary (.WASM) for the engine of about 12 mo and my “game” and its assets in a .PCK of 6 mo. As much as I don’t think it’s possible to compress my assets more, it seems that it’s possible to save some bytes on the WASM by generating a specific export of the engine in which you disable the maximum of features. Nevertheless, this is only a lesser evil because the WASM is a file that compresses very (very) efficiently, going from 12.0 MB to less than 4 MB just with gzip, which is already used during an HTTP fetch when compression is enabled on the server‚óè
